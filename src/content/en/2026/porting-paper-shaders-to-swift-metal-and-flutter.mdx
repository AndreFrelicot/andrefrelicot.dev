Paper Shaders is a collection of WebGL fragment shaders designed to produce visual backgrounds, textures, halftone patterns, gradients, and generative effects through a small web API.

The original project, [paper-design/shaders](https://github.com/paper-design/shaders), is published under the **Apache 2.0** license.

The original project is web-first: GLSL, React, canvas, textures, images, and presets. My goal was different: keep the same visual catalog, but make it usable in native SwiftUI, Flutter, and Jetpack Compose apps through reusable libraries and demo apps that expose every parameter.

The result is split into three ports:

- **PaperShaders** for Swift/Metal, with a MetalKit core and a SwiftUI layer.
- **paper_shaders** for Flutter, using Flutter runtime shaders and a typed Dart API.
- **paper-shaders-android** for Jetpack Compose, using Android `RuntimeShader`, AGSL, and a typed Kotlin API.

This is not only a syntax translation exercise. A shader that compiles is not necessarily a shader that renders the same image. The real work was building a shared contract across four runtimes: WebGL, Metal, Flutter, and Android/Skia.

## The Shared Contract

The most important part of the port is not the strangest shader function. It is the contract around the shader.

Every effect must receive the same families of parameters:

- a stable coordinate space;
- sizing modes such as `cover`, `contain`, and `none`;
- scale, rotation, origin, and offsets;
- a deterministic `frame`;
- speed;
- colors, images, or noise textures depending on the shader;
- named presets.

That contract makes the ports comparable. If `simplex-noise` or `spiral` do not receive the same UV space, origin, and notion of time, the discussion about shader fidelity starts too late: the systems are already different.

On the web, part of that logic naturally lives in the canvas, the vertex shader, and the WebGL environment. In Swift/Metal, Flutter, and Android, that responsibility has to be placed deliberately. The chosen approach is explicit: sizing and uniforms are part of the public API, and presets expose the same mental model across platforms.

```swift
SimplexNoiseView(
  params: .init(
    colors: ["#FF7B00", "#F9FFEB", "#330D82"],
    stepsPerColor: 2,
    softness: 0.12,
    sizing: ShaderSizingParams(fit: .cover, scale: 1)
  )
)
```

```dart
const SimplexNoiseView(
  params: SimplexNoiseParams(
    colors: ['#FF7B00', '#F9FFEB', '#330D82'],
    stepsPerColor: 2,
    softness: 0.12,
    sizing: ShaderSizing(fit: ShaderFit.cover, scale: 1),
  ),
)
```

```kotlin
SimplexNoiseView(
    params = SimplexNoiseParams(
        colors = listOf("#FF7B00", "#F9FFEB", "#330D82"),
        stepsPerColor = 2f,
        softness = 0.12f,
        sizing = ShaderSizing.pattern(scale = 1f),
    ),
    modifier = Modifier.fillMaxSize(),
)
```

Colors are plain hex strings on every platform, exactly as in the upstream React API: the same three strings produce the same render in React, SwiftUI, Flutter, and Compose, with no per-platform color conversion to get wrong.

Initialization code matters. A visual preset is not only a JSON blob: in a real app, you want to copy a Swift, Dart, or Kotlin snippet, paste it into the project, and get the same render.

## Four Languages, Not Four Looks

The original WebGL implementation uses GLSL. The Swift port uses Metal Shading Language. Flutter accepts GLSL as input within the constraints of its Impeller toolchain. Android uses AGSL, a GLSL-like language executed by Skia through `RuntimeShader`.

On paper, the languages look close enough to make the task feel simple. In practice, details matter:

| Topic | WebGL / GLSL | Metal / MSL | Flutter runtime shader | Android / AGSL |
|---|---|---|---|---|
| Fragment coordinates | `gl_FragCoord` | coordinates passed to the fragment | `FlutterFragCoord()` | local `fragCoord` argument |
| Floating modulo | `mod` | `fmod` + helpers | `mod` GLSL | `mod` AGSL |
| Angles | `atan(y, x)` | `atan2(y, x)` | `atan(y, x)` | `atan(y, x)` |
| Bool uniforms | accepted | Metal types | often encoded as floats | encoded as floats |
| Uniform arrays | `vec4 colors[10]` | typed buffers | strict Dart offsets | fixed uniforms and indexing helpers |
| Textures | WebGL samplers | Metal textures | Flutter samplers | child shaders sampled with `.eval()` |

The hard part is not finding an equivalent function. It is preserving implicit conventions: orientation, pixel ratio, interpolation, precision, uniform order, samplers, and edge behavior.

The Flutter port has one advantage: a large portion of the original GLSL remains reusable. But it also has a strict constraint: every uniform has an exact position in the buffer expected by Flutter. A wrong offset does not always break compilation; it can turn a color into a scale, a boolean into a threshold, or an offset into a speed.

The Metal port is more explicit. You get types and native integration, but you pay for a shader-by-shader translation from GLSL to MSL.

AGSL sits somewhere between the two. Its syntax stays close to GLSL, but it is not OpenGL ES or a complete Vulkan pipeline. It is a fragment-style language integrated into Android's Skia rendering pipeline, available through `RuntimeShader` from Android 13 / API 33. Compose can paint it through a `Canvas` and a shader brush, while the upstream vertex sizing logic is inlined into the fragment source.

Textures require special care on Android. Images and the embedded noise texture become child `BitmapShader` inputs; their `.eval()` coordinates are pixels rather than normalized UVs, and AGSL expects premultiplied output colors. Those details compile cleanly when wrong, but produce visibly different renders.

## The Catalog in Six Families

Rather than a shader-by-shader walkthrough, here is the whole catalog grouped by visual family. Every image below is a golden render from the parity suite — each shader's default preset, exactly as the tests see it.

**Gradients** — `mesh-gradient`, `static-mesh-gradient`, `static-radial-gradient`, `grain-gradient`: soft color fields, from an animated color mesh to a radial ramp, plus a version roughened by film-like grain.

Mathematically, a mesh gradient is not a gradient but Shepard interpolation: every pixel averages the color points weighted by inverse distance, `w = 1/(d^3.5 + eps)`, while the points drift along `sin`/`cos` paths. Nothing is drawn — the softness is the average itself. `static-radial-gradient` is the most analytic of the family: it solves a ray–circle intersection (one quadratic per pixel) to produce an off-center radial falloff, and `grain-gradient` quantizes a shape field into a color ramp before roughening it with simplex grain.

**Organic noise** — `simplex-noise`, `perlin-noise`, `neuro-noise`, `smoke-ring`, `gem-smoke`, `warp`: continuous noise put to work — banded contours, speckled fields, glowing filaments, smoke, and marbled distortion.

Three kinds of "noise" coexist here. True simplex (the classic Ashima implementation, two octaves scrolling in opposite directions), true 3D Perlin (corner gradients, quintic interpolation `t^3(t(6t-15)+10)`) stacked as fbm — *fractional Brownian motion*, the same noise summed again at increasing frequency and decreasing amplitude, up to 8 octaves here — and fake: `neuro-noise` is not noise at all but a 15-iteration loop that rotates the frame by one radian per step and accumulates `sin`/`cos`; the interference pattern makes the filaments. The other big idea in the family is domain warping: `warp` distorts the coordinates themselves (`uv.x += s/i * cos(t + 1.5*i*uv.y)`, up to 20 iterations) before drawing plain stripes, and `smoke-ring` computes its fbm in polar coordinates to ripple the radius of a ring. Noise used as a value gives clouds; noise used as a displacement gives marble.

**Polar effects** — `spiral`, `swirl`, `god-rays`: pure `(angle, radius)` territory — concentric spirals, twisted color bands, and rays fanning out from a point.

The whole family fits in one formula: `fract(a*angle + b*f(radius))` after the polar transform. `spiral` takes `f(r) = pow(r, density)` — density tightens the arms; `swirl` takes a negative power, `pow(r, -twist)`, which twists N angular bands into a pinwheel; `god-rays` replaces the radial term with value noise sampled along the angle — noise in the angle is rays. The center is the sensitive spot: as the radius approaches zero the angle becomes unstable, and the `atan` discontinuity at ±π has to be hidden (`god-rays` blends two noise variants across the seam).

**Patterns and geometry** — `dot-grid`, `dot-orbit`, `voronoi`, `metaballs`, `color-panels`, `waves`, `pulsing-border`: repetition and cells — dot lattices, orbiting particles, Voronoi mosaics, merging metaballs, stacked translucent panels, zigzag waves, and a glowing border.

Here space is partitioned before anything is drawn: `floor`/`fract` cut the plane into cells (`dot-grid`), a 3×3 Voronoi loop finds the nearest cell center (`dot-orbit`), and `voronoi` runs a second pass with Inigo Quilez's border metric — the distance to the bisector between neighboring cells — to get clean edges. Each cell receives a hash that varies size, opacity, or color: repetition without uniformity. `metaballs` is a scalar field — a sum of `pow(1 - d/2, p)` falloffs, one per ball — thresholded at 0.4: the blobs merge because the fields add up before the threshold. And `color-panels` hides a real perspective projection, `z = y/(sin(a) - y*cos(a))`, inside a 2D fragment shader.

**Halftone and dithering** — `halftone-dots`, `halftone-cmyk`, `dithering`, `image-dithering`, `heatmap`: the quantization family — a continuous input, procedural or a user image, forced into dots, CMYK separations, dither patterns, or heat levels.

This is quantization against a threshold that is structured in space. `halftone-dots` turns luminance (Rec. 709 weights: `0.2126*R + 0.7152*G + 0.0722*B`) into a dot radius on a grid; `halftone-cmyk` separates the image into inks (`K = 1 - max(R, G, B)`) and screens each channel at its own angle — 15°, 75°, 0°, 45°, the actual print screen angles, which is where the rosettes come from. The two dithering shaders compare the signal against hardcoded 2×2, 4×4, and 8×8 Bayer matrices — `step(0.5, value + threshold - 0.5)`, one bit per pixel. `heatmap` closes the family: a heat scalar runs through a multi-stop color ramp — the lookup table (LUT) of thermal cameras. In every case the eye does the integration: dot size or dot density reads as tone again.

**Materials** — `paper-texture`, `fluted-glass`, `liquid-metal`, `water`: simulated surfaces, usually applied over a user image — paper, ribbed glass, liquid chrome, and water.

No real optics here — only UV displacement and simulated lighting. `fluted-glass` refracts by laterally shifting where the image is sampled, following a periodic per-flute profile (cubic, quadratic, or sine); `water` layers a low-frequency simplex (the waves) with a 6-iteration caustic accumulator, both displacing the UV, and the caustic squared becomes the highlight; `paper-texture` stacks noises (fibers as the gradient magnitude of an fbm, folds as distances to random points, drops as Worley cells), derives a normal by finite differences, and lights it with a plain Lambertian dot product; `liquid-metal` builds its chrome from a `fract()` stripe ramp whose red and blue channels are phase-shifted — chromatic dispersion as a subtraction. The question each pixel answers is never "where does the light go", but "which pixel of the image do I read, and how bright".

## The Small Math Behind Portability

Rather than going back through the shaders one by one, the more useful angle is the small set of mathematical families that power all six groups above, and explain why the same effect can remain portable.

### 1. UV Space

Most shaders start by turning the current pixel into normalized coordinates.

```glsl
vec2 uv = fragCoord / resolution;
vec2 p = uv - 0.5;
```

From there, the shader can move the origin, apply scale, rotate the basis, then feed those coordinates into a visual function. That is the foundation.

Rotation, for example, is not mysterious shader magic. It is a 2D matrix.

```glsl
mat2 rotate(float a) {
  float c = cos(a);
  float s = sin(a);
  return mat2(c, -s, s, c);
}
```

If this transform is not identical across platforms, the shader can be perfectly translated and still produce an offset, flipped, or stretched image.

### 2. Polar Coordinates

Effects such as `spiral`, `swirl`, radial gradients, and several distortions rely on converting `(x, y)` into `(angle, radius)`.

```glsl
float angle = atan(p.y, p.x);
float radius = length(p);
float bands = sin(angle + radius * frequency);
```

This transform is compact, but sensitive. Near the center, the radius approaches zero; the angle becomes unstable; small precision or anti-aliasing differences become visible. This is exactly the kind of shader where a port can look correct at first, then reveal a cross pattern, jagged edge, or strange oscillation on a specific backend.

### 3. Noise

Paper textures, grain, smoke, and organic gradients use noise as a controlled source of variation.

Noise is not only there to make the image dirty. It breaks numerical perfection. A surface that is too regular feels artificial; a bit of noise makes it closer to paper, ink, or material.

In this port, noise is also a practical constraint: some shaders use an embedded noise texture. The same PNG must be loaded with the same sampling conventions, otherwise the render drifts. A repeated 128×128 texture, filtered or sampled differently, can be enough to change the character of a shader.

### 4. Quantization

The `halftone`, `dithering`, and heatmap families do the opposite of continuous noise: they force the render into discrete levels.

```glsl
float stepped = floor(value * steps) / steps;
```

This operation is intentionally harsh. An image becomes a pattern, a color becomes a level, and a gradient becomes a set of bands. It is also the kind of effect that immediately exposes rounding differences between platforms.

### 5. GPU Derivatives

The most subtle part of the port was probably derivative-based anti-aliasing.

In GLSL, `fwidth(x)` estimates how much `x` changes across neighboring pixels. It is very useful for smoothing an edge:

```glsl
float edge = smoothstep(threshold - fwidth(v), threshold + fwidth(v), v);
```

But `fwidth` is tied to GPU execution. It depends on how the backend groups fragments and computes derivatives. For interactive rendering, that is usually what you want. For golden image parity, it is more complicated: two platforms can be visually very close while producing slightly different pixels along edges.

The compromise for tests was to replace some uses with a controlled deterministic derivative when parity required it. Flutter made part of that decision for me: its test runner rejects `fwidth(float)` outright, along with dynamic uniform-array indexing. This is not an artistic rewrite of the shader. It is a way to separate the behavior being tested from the local GPU implementation detail.

## Testing Shaders With Golden Images

The deterministic `frame` changes everything. If a shader depends only on uniforms, textures, and a frame number, it can be rendered offscreen and compared against a reference PNG.

Each preset becomes a visual test case — 120 golden renders across the catalog. This is not perfect: image comparison does not prove that a shader is mathematically identical. But it catches the mistakes that matter for this kind of package:

- wrong uniform order;
- inverted color;
- incorrect rotation or origin;
- missing texture;
- sampling error;
- anti-aliasing drift;
- non-deterministic frame or speed.

For a shader library, the image is the contract.

The Android suite renders the same deterministic cases on a Pixel 7 Pro. All 29 shaders pass the shared comparator, with narrow per-shader allowances for GPU derivatives, texture sampling, and noise functions where exact bytes are not portable.

## The Performance Cost of Precise Math

The first fullscreen runs of the Metal port on real devices were humbling: shaders that animate smoothly in a browser tab were dropping frames as native views. The GPU was not the problem. My parity setup was.

To keep golden renders aligned with WebGL, the Metal pipeline compiles with safe math and precise floating-point functions. Fast math lets the compiler contract and reassociate float operations and substitute approximate `sin`/`cos` — enough to decorrelate hash-based noise from the reference; `perlin-noise` octaves visibly drift. So precise mode is exactly what parity tests need. It is also significantly slower on shaders like `neuro-noise` that accumulate a lot of trigonometry per pixel — and a fullscreen Retina view pushes far more pixels than a typical browser canvas.

The package now treats parity and interactivity as two different rendering modes:

```swift
// Golden parity: precise float functions, fixed resolution.
ShaderRenderOptions.fixed

// Interactive: fast math, plus an internal resolution that
// adapts until the view sustains the target frame rate.
ShaderRenderOptions.adaptive(targetFrameRate: 60)
```

Fast math is the biggest lever; pixel count is the other one. The pixel budget mirrors upstream: a minimum pixel ratio and a cap on total rendered pixels. On top of that, adaptive mode runs a small feedback loop: frame durations are smoothed with an exponential moving average; when the smoothed value exceeds the target by more than about 12%, the internal resolution steps down; once there is comfortable headroom, it slowly steps back up, never below a configurable floor. Fragment-bound shaders scale almost linearly with pixel count, so a small resolution drop — usually invisible on an animated background — buys the frame rate back.

Each view can also report live render metrics (frames per second, drawable size, effective pixel ratio, adaptive scale, math mode), which the demo app displays as an overlay. And to keep the comparison honest, the workbench grew a standalone fullscreen web page that runs the upstream WebGL shaders on the same devices and reports FPS, canvas size, and device pixel ratio. Same shader, same preset, same screen — the only way to know whether the native port is actually slower, or simply rendering more pixels.

Android does not expose an equivalent public fast-math compiler switch for AGSL. The port therefore keeps calculations in `float`/`vec` precision and uses adaptive internal resolution as its main performance lever. It also distinguishes animated effects from static ones: a static shader schedules no frame ticker and redraws only when its uniforms, size, or input image changes. The same static-rendering optimization is now shared by the Swift and Flutter ports.

## Demo Apps

The libraries are not backed only by README files. Each port has a showcase app.

The SwiftUI, Flutter, and Android/Compose apps let you browse all shaders, switch presets, edit uniforms, test sliders, colors, sizing, and copy initialization code. The Android layout adapts between compact phones and wider tablet windows and exposes the same FPS and render-resolution metrics as the other native showcase.

That last detail is intentional. A showcase that displays a render is useful; a showcase that gives you the exact code needed to reproduce it is much more useful.

Shader lists use static thumbnails generated from default presets. I chose pre-rendered assets instead of live rendering every cell: it is more stable, faster, and enough for navigation. Animation stays reserved for the active shader view.

## Takeaway

Porting a shader is not only a syntax translation. It is a conservation problem: preserve a coordinate space, a notion of time, a texture, a rounding behavior, a threshold, and a visual intent.

The differences between GLSL, MSL, Flutter runtime GLSL, and AGSL are manageable. They mostly become visible when you demand parity, not only a good-looking approximation.

That is where the project becomes interesting: a small graphics library becomes a practical lab for render portability. The math stays modest, but it is everywhere: coordinates, rotations, noise, quantization, and derivatives. And precisely because the math is simple, it has to be aligned carefully.
