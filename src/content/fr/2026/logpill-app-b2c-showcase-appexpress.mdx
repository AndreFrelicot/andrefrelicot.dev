LogPill est une application de rappel de prise de médicaments et de suivi de traitement. L'idée est volontairement directe : aider une personne à savoir quels médicaments elle doit prendre aujourd'hui, à enregistrer une dose rapidement, à retrouver son historique et à visualiser son observance dans le temps.

Ce qui m'intéresse ici, c'est surtout l'angle produit. LogPill est une app B2C complète, avec une identité propre, des stores iOS et Android, un site public, des visuels marketing localisés et une base technique proche de ce que peut produire l'offre AppExpress pour un vrai lancement.

## Pourquoi un showcase AppExpress

Les projets AppExpress ne sont pas seulement des prototypes ou des back-offices. L'objectif est de pouvoir livrer un produit exploitable : application mobile, présence web, base de données, authentification si nécessaire, pages légales, assets stores et déploiement.

LogPill permet de montrer ce spectre sur un cas B2C. Le produit doit être simple en apparence, mais il concentre beaucoup de sujets concrets :

- une expérience mobile rapide, utilisable tous les jours ;
- des rappels locaux fiables ;
- une base de données embarquée, disponible sans réseau ;
- des écrans d'analyse lisibles sans devenir médicaux ;
- des captures et textes localisés en anglais, français, espagnol et portugais ;
- un prototype web/SaaS pour visualiser ce que donnerait la synchronisation cloud lorsque le produit devra dépasser le téléphone.

## Le produit

La vue principale donne le planning du jour, le nombre de doses restantes et les prochains rappels. L'utilisateur peut enregistrer une prise, ignorer une dose ou consulter l'historique sans passer par un parcours lourd.

La partie médicaments sert à garder les informations utiles : nom, dosage, forme, fréquence, quantité restante et consignes de prise. Les analyses affichent plutôt des tendances : taux d'observance, série en cours, doses prises ou manquées, comparaison par médicament et suivi des recharges.

## La stack technique

LogPill repose sur les briques AppExpress habituelles : React Native avec Expo pour le mobile, Next.js pour le web, Supabase lorsque le produit a besoin d'une couche compte, stockage ou synchronisation.

| Couche | Choix | Rôle |
|---|---|---|
| Mobile | Expo, React Native, TypeScript | Application iOS et Android avec une base de code commune. |
| Navigation | Expo Router | Routage fichier, tabs et écrans empilés sans configuration lourde. |
| Données locales | WatermelonDB, SQLite | Journal de prises et médicaments accessibles hors ligne. |
| État UI | Zustand | État applicatif léger, sans sur-architecture. |
| Rappels | Expo Notifications | Notifications locales pour les prises planifiées. |
| Localisation | i18next, react-i18next | Interface et assets adaptés aux langues de lancement. |
| Web | Next.js 16, React 19, TypeScript | Site public actif et prototype de dashboard web. |
| Backend web | Supabase, Zod, React Hook Form | Base technique pour authentification, données, validation et formulaires côté dashboard. |

Sur mobile, le coeur de l'expérience est local-first. Une prise doit pouvoir être enregistrée immédiatement, même sans réseau. C'est le genre de détail qui change l'architecture : la base locale n'est pas un cache cosmétique, elle devient la source opérationnelle de l'app.

Le site officiel public est actif sur [logpill.app](https://logpill.app/). La partie dashboard web, synchronisation cloud et mode SaaS existe dans le code et dans les mockups AppExpress comme prototype, mais elle n'est pas active ni déployée dans la version publique de LogPill à ce stade. Les captures ci-dessous documentent cette direction produit, pas une fonctionnalité disponible aujourd'hui.

Dans le code, cette partie web s'appuie sur Next.js, Supabase, Tailwind, Radix UI, Lucide, Framer Motion, React Hook Form, Zod, Resend et React Email.

## Prototype web/SaaS non déployé

## Ce que LogPill valide pour AppExpress

LogPill montre qu'un produit B2C ne se limite pas à "faire une app". Il faut aussi penser la première impression, les stores, les langues, les permissions système, les pages de support, la politique de confidentialité et le modèle de données.
